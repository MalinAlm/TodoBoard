# What we built
We chose to work only with frontend development in order to spend more time on Git collaboration and agile workflows instead of backend and database solutions. Since the project timeframe was limited, we focused on building a simple MVP.
During the project, we completed user story 1 and 2.

# Collaboration in the group

We worked exclusively through mob programming, where one person acted as the driver while the others were navigators, rotating roles regularly.

After our mob sessions, we held mini retros where we reflected on what worked well and what could be improved. We noticed that the collaboration improved over time as we continuously refined our workflow after each retrospective. For example, during the early mob sessions we often got stuck in lengthy discussions. Through our retrospectives we learned to limit these discussions in later sessions and maintain better focus.

Based on lessons learned during our mini retros, we also started doing research before the mob sessions in order to work more efficiently.

# How we worked with Git

All code was developed in feature branches and then merged through pull requests into `dev` and finally from `dev` into `main`. We made regular commits and pushes after each mob session and created pull requests whenever a user story was completed.

One lesson we learned was that merge conflicts do not occur naturally when working only with mob programming. Since we did not want to miss the opportunity to practice resolving a merge conflict together as a new group, we intentionally created one and solved it together.

Naturally we encountered some issues during the project. Because we wanted to avoid committing directly to `main`, we accidentally created an empty `main` branch when setting up the repository and immediately started working in `dev`. Later we discovered that this caused `main` to never be properly published, which we then solved together as a group.

When reviewing the repository's Git history, we believe that both our commit messages and pull requests became significantly clearer over time.

# What worked well

We believe our biggest success factors were the communication within the group, the fun and educational mob programming sessions, our mini retros after each session and the fact that we continuously used version control instead of saving everything until the end.

# What we would have done differently

When we ran out of time for the project, only 2 out of 5 user stories were completed. Looking back, we think we should have either planned a smaller and simpler MVP from the beginning or spent less time on planning and discussions.

Other factors that affected our time management were that we sometimes got stuck in technical discussions and implementation research instead of quickly choosing a solution and moving forward.

We chose to rely entirely on mob programming since part of the assignment was to practice that method. However, we believe we could have saved time on certain tasks if we had also done some individual programming work.

At the same time, we believe that this way of working gave us a better understanding of mob programming and its benefits and uses than if we had worked more individually. Plus, we had a lot of fun during our mob sessions.
